# Implementation Plan: Simplify Auth Model (Remove Doctor Role)

**Branch**: `003-remove-doctor-role` | **Date**: 2026-07-11 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `/specs/003-remove-doctor-role/spec.md`

**Upstream**: 001-better-auth-system, 002-social-oauth-login — this is an
amendment that simplifies the auth model established in 001 and carries
forward in 002.

## Summary

Drop the `doctor` role entirely. Reduce `User.role` to two values
(`user`, `admin`). Replace the `DoctorProfile` table (1:1 with `User`)
with a standalone `Doctor` table managed by admins via full CRUD.
Remove the doctor-only route guard, the doctor approval flow, and the
"patient" terminology. Add a `PATCH /api/admin/users/:id/role`
endpoint with a last-admin guard.

The work is concentrated in:
- `prisma/schema.prisma` — drop `DoctorProfile`, add `Doctor`, change
  `User.role` default.
- `src/auth/auth.ts` — change role default, reject non-`user`/`admin`
  role values in the `databaseHooks.user.create.before` callback,
  remove the doctor-profile auto-creation hook.
- `src/admin/admin.controller.ts` and `admin.service.ts` — replace
  doctor-profile approval endpoints with doctor CRUD.
- `src/common/guards/doctor-approved.guard.ts` — delete the file.
- `src/auth/auth.controller.ts` — remove `/api/doctors/test-route`.
- `src/auth/auth.service.ts` — remove the doctor-profile enrichment
  branch in `enrichSessionUser`.
- `src/common/interfaces/session.interface.ts` — drop
  `DoctorProfileSummary` and `SessionUser.doctorProfile`.

No new third-party dependencies. No new database tables beyond the
`Doctor` table (which replaces `DoctorProfile`).

## Constitution Check

*GATE: Must pass before Phase 0 research. Re-check after Phase 1 design.*

| Principle | Status | Notes |
|-----------|--------|-------|
| I. Feature-Module Architecture | PASS | All changes live in the existing `src/auth/` and `src/admin/` modules. The new `Doctor` entity is owned by the `admin` module. No new top-level module. |
| II. Domain-Event Decoupling | PASS | No new cross-module communication. The role change and doctor CRUD are admin-internal. |
| III. Single-Clinic Identity | PASS (amended) | This feature is a **strict simplification** of Principle III: doctors are no longer modeled as a 1:1 relation to `User` at all. They are standalone records. The "single-clinic identity" is preserved — there is still no multi-clinic modeling. The constitution should be amended in a follow-up to reflect the new shape (out of scope here, but flagged). |
| IV. Transactional Data Integrity | PASS | The role-change endpoint is a single-row update. The doctor CRUD is single-row. No resource-claiming operations. |
| V. Better Auth Schema Sovereignty | PASS | The `User` schema is unchanged at the Better Auth level. The role default change is in `additionalFields`. The `DoctorProfile` removal is a feature-specific concern (it was a custom model, not Better Auth core). |
| VI. Auth & Access Control | PASS | Session-based auth is unchanged. The existing `RolesGuard` is reused. The last-admin guard is a server-side check in the controller. No re-implementation of Better Auth primitives. |
| VII. Phased Delivery | PASS | This is an amendment to Phase 1 (auth). It is a strict simplification — no new scope. |

**Re-evaluation after Phase 1 design**: All gates still pass. The
data model document explicitly notes that the `Doctor` table is
standalone (no FK to `User`), which is a cleaner fit for Principle III
than the previous `DoctorProfile` 1:1 relation.

All gates pass. **Note**: A follow-up constitution amendment is
recommended to update Principle III wording (from "1:1 relation to
`User`" to "standalone doctor record with no auth relationship"),
but that amendment is out of scope for this feature.
