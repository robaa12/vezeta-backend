# Data Model: Social Login (Google & Facebook via Better Auth)

**Feature**: 002-social-oauth-login
**Date**: 2026-07-11
**Upstream**: 001-better-auth-system (data model inherited)

## Overview

This feature adds no new database tables. The `Account` table generated by
Better Auth in feature 001 already supports social provider accounts
(`providerId = "google" | "facebook"`). The work in this feature is purely
configuration + controller endpoints + business rules that operate on
existing tables.

## Entities Affected

The following entities from feature 001 are touched by this feature. Their
schema is unchanged; only their behavior and the rules around them are
extended.

### User (Better Auth core, unchanged schema)

| Field | Type | Notes for this feature |
|-------|------|------------------------|
| id | String (cuid) | PK |
| name | String | Populated from social profile on signup; user may edit later |
| email | String (unique) | Verified at signup because the provider verified it |
| emailVerified | Boolean | Set to `true` on social signup |
| phoneNumber | String? (unique) | Not populated by social signup in v1 |
| phoneNumberVerified | Boolean | Not set by social signup in v1 |
| role | String (default "patient") | Social signup always creates role `patient`; no `doctor` option |
| isActive | Boolean | If `false`, social sign-in is rejected (FR-014) |
| image | String? | Populated from social profile `picture` URL on signup |
| createdAt / updatedAt | DateTime | Auto |

**New behavior**:
- On social signup: `role` is forced to `"patient"`; the signup endpoint
  never accepts role input from the social flow (FR-004). A user who wishes
  to be a doctor must use the email/password flow from feature 001.
- On social signup: `emailVerified` is set to `true` because the provider
  has already verified the email (FR-003).
- On social signup: `name` and `image` are populated from the social profile
  (`profile.name` / `profile.picture`).
- On any social sign-in attempt (existing or new): the `account.accountLinking`
  hook rejects the operation with "account deactivated" if `isActive = false`
  (FR-014).

### Account (Better Auth core, unchanged schema)

The `Account` table is the central entity for this feature. One row per
linked authentication method.

| Field | Type | Notes for this feature |
|-------|------|------------------------|
| id | String (cuid) | PK |
| userId | String (FK → User) | Owner of the linked account |
| accountId | String | Provider's user id (e.g. Google's `sub` claim) |
| providerId | String | `"credential"`, `"google"`, or `"facebook"` |
| accessToken | String? | OAuth access token (managed by Better Auth) |
| refreshToken | String? | OAuth refresh token (managed by Better Auth) |
| accessTokenExpiresAt | DateTime? | When the access token expires |
| refreshTokenExpiresAt | DateTime? | When the refresh token expires |
| scope | String? | Space-separated list of granted scopes |
| idToken | String? | OIDC ID token (Google) |
| password | String? | Hashed password — present only for `providerId = "credential"` |
| createdAt / updatedAt | DateTime | Auto |

**New business rules (enforced in our custom endpoints, not at DB level)**:

1. **FR-010 — Max one per provider**: For any given `userId`, there is at
   most one `Account` row with `providerId = "google"` and at most one with
   `providerId = "facebook"`. The unique combination `(userId, providerId)`
   is enforced by the link endpoint.

2. **FR-013 — Last sign-in method protection**: Before unlinking a social
   `Account`, the controller counts the user's remaining sign-in methods:
   - One credential `Account` with `password IS NOT NULL` counts as one method.
   - Each social `Account` (Google, Facebook) counts as one method.
   - The unlink is rejected if removing the target would leave the user with
     zero sign-in methods.

3. **FR-009 / FR-010 — Email match for explicit linking**: Before allowing an
   explicit link via `POST /api/auth/link-social`, the controller compares
   the OAuth profile's verified email to `User.email` of the currently
   authenticated user. Mismatch returns 422 with `"email_mismatch"`.

4. **FR-008 — Verified email required for any social signup/link**: If the
   provider returns no email or the email is not marked verified, Better
   Auth's account-linking hook rejects the request.

**Unique constraint implication**: Better Auth's stock behavior already
enforces a unique `(providerId, accountId)` constraint at the DB level
(provider-side user id is unique per provider). Our custom code does not
add new unique indexes; the per-user-per-provider rule (rule 1 above) is
enforced in the controller's pre-check.

---

### Session (Better Auth core, unchanged schema)

No new behavior. Sessions created by social sign-in are indistinguishable
from sessions created by credential sign-in — same cookie, same TTL, same
`@Session()` injection.

---

### DoctorProfile (unchanged schema, no creation in this feature)

| Field | Type | Notes for this feature |
|-------|------|------------------------|
| id, userId, status, approvedById, approvedAt, createdAt, updatedAt | ... | Unchanged |

**New behavior**:
- Social signup NEVER creates a `DoctorProfile` row in v1. The role is
  always `patient`; a user who wants doctor standing must use the
  email/password flow (FR-004 / SC-004).
- A Super Admin may promote a social-signup `patient` to `doctor` out of
  band and create the `DoctorProfile` record manually — this is not a
  self-service path and is not exposed as an endpoint in this feature.

## Migration Strategy

**No migration is required.** The `Account` table from feature 001 already
has all columns Better Auth's social providers need. Adding this feature
is a pure code change:

1. Add `socialProviders.google` and `socialProviders.facebook` blocks to
   `src/auth/auth.ts` (existing file from feature 001).
2. Add `accountLinking: { enabled: true, trustedProviders: ["google", "facebook"] }`
   to the same `auth.ts`.
3. Add an `account.accountLinking` hook in `auth.ts` that rejects deactivated users.
4. Add new endpoints in `src/auth/auth.controller.ts` (existing file):
   - `POST /api/auth/link-social` (FR-009/FR-010)
   - `DELETE /api/auth/social-accounts/:provider` (FR-011/FR-012/FR-013)
5. Extend the existing `GET /api/auth/me` (or `/api/auth/whoami`) response
   with a `linkedSocialProviders` field (FR-015).
6. Add `GOOGLE_CLIENT_ID`, `GOOGLE_CLIENT_SECRET`, `FACEBOOK_CLIENT_ID`,
   `FACEBOOK_CLIENT_SECRET` env vars to `.env.example`.

## Validation Rules (consolidated)

| Entity.Field | Rule |
|--------------|------|
| User.email | Unique across all users (existing from feature 001) |
| User.emailVerified | `true` immediately after any successful social signup/link |
| User.role | `"patient"` for any social signup; never `"doctor"` from this flow |
| User.isActive | Social sign-in rejected if `false` (FR-014) |
| Account.userId | Must reference an existing User (FK, existing) |
| Account.providerId | One of: `"credential"`, `"google"`, `"facebook"` (FR-001/FR-002) |
| Account.(userId, providerId) | At most one row per user per provider (FR-010, app-level check) |
| Account.password | Non-null only when `providerId = "credential"` (existing from feature 001) |
| Account.accessToken | Required when `providerId` is a social provider (managed by Better Auth) |
| Account.email (in OAuth profile) | Must be present and verified (FR-008) |
