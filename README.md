# Vezeeta Backend

NestJS + Prisma + PostgreSQL backend with **Better Auth** for session-based
authentication. Powers a doctor appointment booking platform with two roles:
`user` (any consumer) and `admin` (Super Admin, seeded). Doctors are a
separate domain entity (standalone CRUD records) managed by admins — they are
not users of the platform and cannot log in.

## Tech Stack

- **NestJS 11** (TypeScript, ESM)
- **Prisma 7** ORM + PostgreSQL
- **Better Auth 1.6** with `@better-auth/prisma-adapter`
  - `emailOTP` plugin (email verification & password reset)
  - `phoneNumber` plugin (SMS verification, phone sign-in, phone password reset)
  - **Social providers** — Google and Facebook (configurable via env vars)
- **@thallesp/nestjs-better-auth** NestJS adapter
- **class-validator** / **class-transformer** for DTOs
- **Jest** for unit + e2e tests

## Setup

```bash
# 1. Install dependencies (uses Node.js 22+, ESM)
npm install

# 2. Configure environment
cp .env.example .env
# Edit .env: set DATABASE_URL, BETTER_AUTH_SECRET, etc.

# 3. Generate Prisma client
npm run db:generate

# 4. Run migrations (requires a running PostgreSQL)
npm run db:migrate

# 5. Seed the Super Admin
npm run db:seed
```

| Method | Path                       | Auth     | Description                          |
|--------|----------------------------|----------|--------------------------------------|
| GET    | `/api/me`                  | session  | Current user profile + linkedSocialProviders |
| GET    | `/api/health`              | public   | Health check                         |

## Public Doctor Endpoints (Module 2)

Anonymous-accessible endpoints for the patient-facing doctor catalog. No
authentication required. See `specs/004-doctor-search/` and
`specs/005-doctor-categories/` for the full specs.

| Method | Path                       | Description                                       |
|--------|----------------------------|---------------------------------------------------|
| GET    | `/api/doctors`             | Paginated, filterable list of ACTIVE doctors      |
| GET    | `/api/doctors/:id`         | Public doctor profile (404 for DEACTIVATED doctor or category) |
| GET    | `/api/categories`          | ACTIVE categories sorted alphabetically (case-insensitive) |

The listing accepts `categoryId` (FK equality, requires the category to be
ACTIVE), `search` (case-insensitive substring on name + category.name),
`page`, `pageSize` (1-100). Each doctor record includes a `category: { id,
name }` object (no free-text `specialty` field — the legacy column was
dropped in feature 005). Responses include `Cache-Control: public,
max-age=N` headers (60 for the listing, 300 for the profile, 300 for the
categories dropdown) and are rate-limited per IP via `@nestjs/throttler`
(60/120/60 requests per minute).

### Better Auth managed routes

- `POST /api/auth/sign-up/email` — create account with role
- `POST /api/auth/sign-in/email` / `sign-in/phone-number` — login
- `POST /api/auth/sign-out`
- `POST /api/auth/email-otp/send-verification-otp` / `verify-email`
- `POST /api/auth/phone-number/send-otp` / `verify`
- `POST /api/auth/email-otp/request-password-reset` / `reset-password`
- `POST /api/auth/phone-number/request-password-reset` / `reset-password`
- `GET /api/auth/sign-in/social?provider=google|facebook` — start social OAuth
- `GET /api/auth/callback/google` / `callback/facebook` — OAuth callbacks

### Custom social auth routes

| Method | Path                                  | Auth    | Description                                          |
|--------|---------------------------------------|---------|------------------------------------------------------|
| POST   | `/api/auth/link-social`               | session | Initiate linking a Google/Facebook account           |
| DELETE | `/api/auth/social-accounts/:provider` | session | Unlink a previously linked Google/Facebook account   |

> (`/approve`, `/reject`, `/suspend`, `?status=PENDING`) have been
> **removed** in feature `003-remove-doctor-role`. Doctors are added
> directly to the platform by admins and are immediately `ACTIVE`; no
> approval gate is needed.

## Architecture

This project follows the Vezeeta constitution:

- **Feature-module layout** — each domain has its own module under `src/`
- **Better Auth schema sovereignty** — auth tables generated, DoctorProfile is separate
- **Session-based auth** — HTTP-only cookies, no JWTs
- **Doctor approval gate** — `DoctorApprovedGuard` checks `DoctorProfile.status === "APPROVED"`
- **Role-based access** — `RolesGuard` + `@Roles()` decorator

See `specs/001-better-auth-system/` for the full feature spec, plan, data model,
and quickstart validation guide.

## ⚠️ Data migration note for `003-remove-doctor-role`

The schema changes in `prisma/migrations/20260712000000_remove_doctor_role/`
are **schema-only** (drop `DoctorProfile`, add `Doctor`, change `User.role`
default). On a **fresh database** they are sufficient.

On a **populated database** (e.g. production), run the following data
migration **before** applying the schema migration:

```sql
-- Coerce any legacy role values to the new enum
UPDATE "user" SET role = 'user' WHERE role IN ('patient', 'doctor');

-- Convert existing DoctorProfile rows to standalone Doctor rows.
-- The exact mapping depends on production data shape; example:
-- INSERT INTO "doctor" (id, name, specialty, status, "createdAt", "updatedAt")
--   SELECT dp.id, u.name, '<migrated-specialty>', 'ACTIVE', dp."createdAt", dp."updatedAt"
--   FROM "doctor_profile" dp JOIN "user" u ON u.id = dp."userId";
```

The schema migration will then drop `doctor_profile`. Do not run the
schema migration until the data migration has been verified on a
staging copy.
